# Git Hooks 是什么

Git Hooks 是：

>   Git 在特定操作前后自动执行的脚本。

比如：

-   提交前
-   提交信息写完后
-   推送前

这些脚本放在：

```text
.git/hooks/
```

------

## 6.1 Pre-commit Hook

这是：

>   在你执行 `git commit` 之前自动运行。

示例脚本做了什么？

```bash
npm run lint
npm test
```

意思是：

1.  先做代码规范检查
2.  再跑测试
3.  只要有一个失败
4.  就拒绝提交

效果是：

```text
代码不合格 → 不允许 commit
```

这是本地“第一道质量门”。

------

## Commit-msg Hook

这是：

>   检查你的提交信息格式。

例如强制要求：

```text
feat(auth): 添加登录功能
```

而不是：

```text
改了一些东西
```

它可以配合：

-   commitlint
-   gitlint

实现提交规范统一。

------

# 二、标签管理（Tag）

标签的作用是：

>   给某个提交打一个“版本标记”。

比如：

```bash
git tag -a v1.0.0 -m "版本 1.0.0"
```

意思是：

```text
这个 commit = 版本 1.0.0
```

------

## 为什么要用标签？

因为分支会移动：

```text
main → 不断前进
```

但标签是固定的：

```text
v1.0.0 → 永远指向那个提交
```

发布版本时非常重要。

------

## 轻量标签 vs 附注标签

### 轻量标签

```bash
git tag v1.0.0
```

只是一个指针。

------

### 附注标签（推荐）

```bash
git tag -a v1.0.0 -m "版本 1.0.0 发布"
```

它是一个完整对象：

-   有作者
-   有时间
-   有说明
-   可以签名

团队开发应使用附注标签。

------

# 三、语义化版本（SemVer）

遵循：

>   主版本.次版本.修订号

例如：

```
v1.2.3
```

规则是：

-   主版本变化 = 破坏性修改
-   次版本变化 = 新功能
-   修订号变化 = Bug 修复

这是现代软件版本标准。

------

# 四、自动化检查（CI/CD 集成）

这是“服务器级别的质量控制”。

当你提交 PR 时：

平台自动：

-   编译
-   测试
-   检查代码规范
-   检查提交格式
-   检查安全漏洞

在：

-   GitHub
-   GitLab

都可以配置。

------

# 五、推荐工具解释

### commitlint

检查：

```text
Commit Message 是否符合规范
```

------

### husky

用来管理 Git Hooks 的工具。

因为：

`.git/hooks` 目录不会被 Git 版本控制。

husky 可以把 hook 管理纳入项目。

------

### prettier

自动格式化代码。

防止：

-   空格混乱
-   缩进不一致
-   团队风格不统一

------

### eslint

JavaScript 静态代码分析工具。

用于：

-   发现潜在错误
-   强制代码规范

## 五、团队协作工作流

## 六、高级实践与工具

### 6.1 Git Hooks 使用

#### Pre-commit Hook

在提交前自动检查代码质量。

**示例：`.git/hooks/pre-commit`**

```bash
#!/bin/sh
# 运行代码检查
npm run lint
if [ $? -ne 0 ]; then
    echo "Lint 检查失败，请修复后再提交"
    exit 1
fi

# 运行测试
npm test
if [ $? -ne 0 ]; then
    echo "测试失败，请修复后再提交"
    exit 1
fi
```

#### Commit-msg Hook

检查 Commit Message 格式。

**使用工具**：

- [commitlint](https://commitlint.js.org/)：检查提交信息格式
- [gitlint](https://jorisroovers.com/gitlint/)：Git 提交信息检查工具

### 6.2 标签管理

#### 标签类型

1. **轻量标签**：指向特定提交的引用

    ```bash
    git tag v1.0.0
    ```

2. **附注标签**：包含标签信息的完整对象

    ```bash
    git tag -a v1.0.0 -m "版本 1.0.0 发布"
    ```

#### 标签命名规范

- **语义化版本**：遵循 [SemVer](https://semver.org/lang/zh-CN/) 规范
    - 格式：`v主版本号.次版本号.修订号`
    - 示例：`v1.0.0`、`v1.2.3`、`v2.0.0-beta.1`

- **标签前缀**：使用 `v` 前缀（如 `v1.0.0`）

#### 标签操作

```bash
# 创建标签
git tag -a v1.0.0 -m "版本 1.0.0"

# 推送标签
git push origin v1.0.0

# 推送所有标签
git push origin --tags

# 删除标签
git tag -d v1.0.0
git push origin :refs/tags/v1.0.0
```

### 6.3 自动化检查

#### CI/CD 集成

在 CI/CD 流程中集成以下检查：

1. **代码质量检查**：
    - Linter 检查
    - 代码覆盖率检查
    - 安全检查

2. **Commit Message 检查**：
    - 格式验证
    - 类型验证

3. **自动化测试**：
    - 单元测试
    - 集成测试
    - E2E 测试

#### 推荐工具

- **commitlint**：Commit Message 格式检查
- **husky**：Git Hooks 管理
- **prettier**：代码格式化
- **eslint**：代码质量检查
