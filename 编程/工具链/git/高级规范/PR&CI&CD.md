# PR&CI&CD

很好，这两个词是现代软件协作的核心概念。我们讲清楚它们在实际开发中到底“干什么”。

------

# 一、PR 是什么？

PR = **Pull Request**

在 GitHub 上叫 Pull Request，
在 GitLab 上叫 Merge Request（MR）。

------

## 1️⃣ PR 的本质

>   “我写完代码了，请求你把我的分支合并到主分支。”

比如：

```text
main   ←  feature/login
```

你做完登录功能后，不会直接：

```bash
git push origin main
```

而是：

1.  把代码推到 feature 分支
2.  在平台上创建 PR
3.  让别人审核
4.  审核通过后再合并

------

## 2️⃣ PR 解决什么问题？

-   防止直接污染主分支
-   让代码变更可讨论
-   留下审查记录
-   可以自动触发测试
-   可以回滚

在团队开发中，PR 是“质量控制关卡”。

------

## 3️⃣ PR 的流程（实际工作中）

1.  新建分支

```bash
git checkout -b feature/login
```

1.  开发完成

```bash
git push origin feature/login
```

1.  在 GitHub 上点：

```
Create Pull Request
```

1.  等待：
    -   代码审查
    -   自动测试
    -   合并

------

# 二、CI 是什么？

CI = **Continuous Integration（持续集成）**

它的意思是：

>   每次代码提交后，自动帮你做检查。

------

## CI 会自动做什么？

通常包括：

-   自动编译
-   自动运行单元测试
-   静态代码检查
-   格式检查
-   安全扫描

如果失败：

```text
❌ 不允许合并
```

------

## 举个实际例子

你提交 PR 之后：

平台自动运行：

-   编译项目
-   跑 500 个测试用例
-   检查是否有警告

如果有一个测试失败：

```text
PR 状态 = Failed
```

必须修复后才能合并。

------

# 三、CD 是什么？

CD = **Continuous Delivery / Continuous Deployment**

有两个意思：

-   Continuous Delivery = 持续交付
-   Continuous Deployment = 持续部署

------

## CD 做什么？

当 PR 合并到 main 后：

系统自动：

-   打包
-   构建镜像
-   发布到服务器
-   更新线上服务

------

# 四、CI/CD 审查是什么意思？

你之前那段话说：

>   必须通过 CI/CD 检查

意思是：

1.  代码必须编译成功
2.  测试必须全部通过
3.  自动检查必须无错误
4.  部署流程必须成功

否则：

```text
不能合并到 main
```

------

# 五、用一句话概括

| 名称 | 本质         |
| ---- | ------------ |
| PR   | 人工审核关卡 |
| CI   | 自动技术检查 |
| CD   | 自动发布     |

------

# 六、为什么现代团队必须用这些？

如果没有：

-   有人 push 破坏代码
-   编译都不过
-   线上直接挂
-   没人发现是谁改的

CI/CD 是“自动化质量保障系统”。

------

# 七、结合你的情况

你现在是个人开发：

-   ESP32
-   Linux
-   嵌入式

你暂时不一定需要完整 CI/CD。

但如果将来：

-   开源项目
-   团队协作
-   商业项目

PR + CI 是必须技能。

------

# 最终总结

PR = “请把我的代码合进去”

CI/CD 审查 = “自动帮你检查代码质量并部署”
