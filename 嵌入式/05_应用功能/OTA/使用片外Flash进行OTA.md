# 使用片外Flash进行OTA

假设一个单片机拥有512KB可以直接执行代码的片内Flash，和8MB的不可以直接执行代码片外Flash。

如果进行OTA升级，一般有两种思路：

-   以ESP32为代表的 `Bootloader + App A + App B` 双应用分区模式。
-   



编写 `Bootloader + App` 分区程序，要求：

1.  Bootloader检查标志位判断是否需要进行OTA升级：
    -   如果是：从片外Flash指定地址读取升级固件，分片写入App分区，然后更新标志位，从App分区启动。
    -   如果不是：直接从App分区启动。
2.  App必须拥有向片外Flash写入的能力，并且使用片内Flash的最后一两页作为标志位分区，存放标志位。
3.  App分区拥有接收新的固件的能力，并且写入片外Flash，固件来源包括但不限于串口。
4.  App分区必须包含实际的除了OTA之外的应用代码。



OTA升级流程：

1.  MCU启动，首先启动Bootloader，Bootloader检查标志位，没有OTA升级标志，引导启动App。
2.  App执行应用代码，并且准备接收新的固件。
3.  App开始接收新的固件，并且将新的固件分片写入片外Flash，完成后更新标志位。
4.  复位，重启。
5.  再次启动Bootloader，Bootloader检查标志位，有OTA升级标志，开始把片外Flash的程序分片写入片内，清除标志位，复位重启。
6.  升级完毕。



上电 → Bootloader

-   无升级标志 → 跳转 App
-   有升级标志 → 进入升级流程

App运行

-   接收新固件 → 写入片外 Flash
-   完成后：标记“待升级” → 重启

Bootloader执行升级

-   标记“升级中”
-   片外 → 片内 分片写入
-   校验成功 → 标记“正常” → 重启
-   校验失败 → 进入恢复模式

升级完成 → 运行新 App



边接收边计算：  
每收到一个字节，更新 CRC 计算器，同时将该字节写入 SPI Flash。  
全部接收完成后，将最终 CRC 值与预设的校验值（如附加在固件末尾或通过其他通道获取）比较。  
若一致则标记固件有效，否则丢弃。
